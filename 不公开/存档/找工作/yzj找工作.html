  1. Message Queues:

    • By using message queues (like RabbitMQ, Kafka, or Amazon SQS), we can decouple our services and allow for asynchronous processing. For example, when a user attempts to book a car, the request can be placed into a queue. A separate service can then process these bookings, ensuring that cars aren’t double-booked and handling any necessary error conditions. This approach can prevent immediate system overloads during spikes in demand.

  2. Auto-scaling:

    • Employ cloud-based auto-scaling solutions, such as those offered by AWS, Google Cloud, or Azure. With proper monitoring and thresholds, the system can automatically spawn new instances of services when demand increases and reduce instances when demand decreases.

  3. Load Balancing:

    • Deploying a load balancer can efficiently distribute incoming traffic across multiple servers, ensuring that no single server becomes a bottleneck. This can prevent server overloads and provide a seamless experience for the end user.

  4. Caching:

    • Implement caching mechanisms (using systems like Redis or Memcached) to reduce the number of direct hits to the database. For frequently accessed data, such as available cars in popular locations, caching can greatly reduce the system’s response time and reduce database load.

Centralized Database

Even in distributed systems, using a centralized database that supports ACID (Atomicity, Consistency, Isolation, Durability) properties can ensure data consistency. Systems like relational databases (e.g., PostgreSQL, MySQL) enforce these properties.

Locking Mechanism:
When a user tries to book a car, we can “lock” that car’s record in the database. This means that for a short period, no other user can book the same car. If the booking is completed, the car’s status is updated to “unavailable”. If the user doesn’t complete the booking, the lock is released, and the car is available again.

Immediate Feedback:
When a user books a car, immediately update the database and reflect this change to all users. This way, if another user tries to book the same car, they’ll be informed right away that it’s no longer available.

车辆查询功能

Location-based Searching:
Most modern databases provide support for geo-spatial queries. We could store the location of each car using coordinates (latitude and longitude) and allow users to search based on their current location or a provided address.

Indexing:
For efficient querying, it’s essential to have appropriate indexes. In this case, an index on the location coordinates would be beneficial for rapid searches. Similarly, indexing cars based on their availability status (available vs. rented) can speed up the search for available cars.

Filtering:
Beyond just location, users might have specific requirements like car type, brand, or price range. Ensuring indexed fields for these common filters will allow for more efficient searches.

Pagination:
Instead of returning all available cars in a large area, you can implement pagination to return a limited number of results at a time, allowing the user to “load more” if they wish to see more options.

High-Level Design

How would you design a system for a car rental company that operates in multiple cities?

  1. Basic Components:

    • User System: Manages customer details and authentication.
    • Vehicle Inventory System: Tracks the cars available in various cities.
    • Booking System: Allows users to reserve and cancel cars.
    • Payment System: Handles user payments.

  2. Database Design:

    • User Database: Stores user details.
    • Vehicle Database: Keeps track of the status of each car, such as whether it’s rented and when it’s available next.
    • Booking Database: Stores reservation details.

  3. Multi-City Operation:

    • Data can be partitioned based on cities, with each city having its own subset of data.

  4. Security:

    • Ensure safe user authentication.
    • Encrypt data during transmission and storage.

  5. User Experience:

    • Provide an intuitive interface for users to easily find and book cars.

  6. Monitoring:

    • Use basic tools to monitor the health of the system.

  7. Problem Identification: Initially, using server monitoring tools, we observed that during peak user query times, both the server’s CPU and memory usage were high.

  8. Solution Consideration: We thought that if we could handle user query requests in a distributed manner instead of all at once, it might improve efficiency. It’s like a supermarket opening multiple checkout counters to manage the flow of customers, reducing the pressure on each counter.

  9. Introducing RabbitMQ: To implement this idea, we decided to use a tool called RabbitMQ. Its role is like a middleman, collecting user query requests first and then distributing them one by one for the server to process.

  10. Results: After integrating RabbitMQ, our query rate improved from 5,000 times per second to 30,000 times per second. Users were no longer frustrated by long waiting times.

From this experience, I learned that sometimes simple strategies and tools can bring significant improvements.
